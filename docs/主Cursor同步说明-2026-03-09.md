## 这次要同步的重点

本次排查和修复集中在 `src-tauri/src/lib.rs`，目标是解决：

- “启动当前 Agent 网关”失败但界面看不到真实错误
- “一键启动所有 Gateway”看起来只启动一个
- QQ 渠道插件安装没问题，但 gateway 运行配置生成错误
- 旧 `active_telegram_instance` / 旧 gateway 残留会把失效 gateway 自动补回来
- Windows 计划任务模式返回成功，但 gateway 实际并未真正监听端口

## 已确认的根因

### 1. 网关真实失败原因被吞掉

原始错误是：

```text
Gateway install failed: Error: schtasks create failed: 拒绝访问
```

但旧逻辑吞掉了这层错误，后面只剩：

```text
Gateway service missing
```

### 2. 批量启动并不是“只启动一个”

真实情况是：

- `gw-agent-1` 可以启动
- `gw-telegram-tg-main` 是旧残留，启动失败

所以结果表现成“只有一个成功”。

### 3. QQ 插件命令本身没错

修复中心实际执行的是：

```text
openclaw plugins install @sliverp/qqbot@latest
```

QQ 打包来源也已明确：

- 运行时插件包名固定映射为 `@sliverp/qqbot`
- 发布打包前会执行 `scripts/prepare-bundled-extensions.ps1`
- 该脚本会优先从当前 `OPENCLAW_STATE_DIR/extensions/qqbot` 收集已验证可用的 QQ 插件目录
- 然后复制到 `src-tauri/resources/bundled-extensions/qqbot`
- 最终随 Tauri 安装包一起分发，供首次安装或离线场景直接回填到运行目录

问题不在命令，而在后续 gateway 配置生成：

- 插件渠道被错误写进 `openclaw.json.channels`
- 这会触发 `unknown channel id`
- 部分 gateway 的 `state_dir/extensions` 没同步对应插件目录

### 4. 旧 gateway 会被配置残留自动补回

罪魁祸首是运行时配置归一化逻辑会继续信任：

- `active_telegram_instance`
- `active_channel_instances`
- 旧 `gateways`
- 旧 `channel_routes`

只要这些字段残留了失效实例，就可能把旧 gateway 再生成回来。

### 5. 最新确认：网关本体没坏，坏的是计划任务启动链路

实机验证结果：

- 当前运行时 gateway 为 `gw-agent-main`、`gw-agent-1`
- 两者都能通过前台命令 `openclaw gateway` 成功监听端口
- 但通过 `openclaw gateway start` / `schtasks /Run` 触发后，任务虽然返回成功、`LastTaskResult = 0`
- 实际端口不监听，任务状态长期停留在 `Queued`

所以现在的核心根因已经进一步收敛为：

- OpenClaw gateway 进程本体是正常的
- Windows 任务计划程序模式不可靠，导致桌面端误判“已启动”

## 本次代码改动

### 1. `start_gateway_instance()`

- 不再忽略 `ensure_gateway_service_installed()` 的失败
- 遇到 `Gateway service missing` / `schtasks` / `系统找不到指定的文件` 等输出时，直接按失败处理
- 界面能拿到真实错误，而不是被二次掩盖

### 2. gateway 运行配置生成逻辑

- `openclaw.json.channels` 只保留 OpenClaw 内置渠道
- QQ 这类插件渠道不再写进 `openclaw.json.channels`
- 插件渠道改为走 `channels.json + plugins.entries + extensions`

### 3. gateway 插件目录同步

- 新增“按当前 gateway 实际需要”同步插件目录的逻辑
- 不再把整个 `extensions` 全量复制进每个 gateway 的 `state_dir`
- QQ 对应的来源目录已固定为打包资源里的 `bundled-extensions/qqbot`

### 4. 运行时配置归一化逻辑

新增校验：

- 只有真实存在且启用的 Telegram/渠道实例，才允许自动生成或补回 gateway
- 无效的 `active_telegram_instance` 会被清空
- 无效的 `active_channel_instances` 会被剔除
- 旧 gateway 如果绑定实例已失效，不再继续保留旧 `gateway_id`

### 5. `start_gateway_instance()` 增加任务计划失败兜底

新增逻辑：

- 先照常执行 `openclaw gateway start`
- 如果命令返回成功，但几秒内目标端口没有监听
- 自动回退为直接执行 `state_dir/gateway.cmd`
- 这样“启动当前网关”和“批量启动网关”都能绕过计划任务 `Queued` 但不真正运行的问题
