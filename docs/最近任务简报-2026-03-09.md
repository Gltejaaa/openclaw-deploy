# OpenClaw 网关排查简报

时间：2026-03-09

## 核心结论

- 当前实际安装目录：`C:\Users\Administrator\openclaw`
- 当前实际配置目录：`C:\Users\Administrator\openclaw\.openclaw`
- 实际可执行文件：`C:\Users\Administrator\openclaw\openclaw.cmd`
- 当前运行时有效网关：`gw-agent-main`、`gw-agent-1`
- 旧残留网关：`gw-telegram-tg-main`（已清理）
- 旧系统级残留任务：`OpenClaw Gateway`

## 已确认的问题

### 1. 网关启动失败真实原因

原始报错：

```text
Gateway install failed: Error: schtasks create failed: 拒绝访问
```

之前代码吞掉了这层错误，界面只会看到后续的：

```text
Gateway service missing
```

### 2. “一键启动所有 Gateway 只启动一个”的真实原因

- `gw-agent-1` 可以启动
- `gw-telegram-tg-main` 是旧残留，无法正常启动
- 所以批量启动不是只执行了一个，而是一个成功、一个失败

### 3. QQ 渠道问题的真实原因

修复中心安装 QQ 插件命令本身没问题，代码实际执行的是：

```text
openclaw plugins install @sliverp/qqbot@latest
```

真正的问题在于网关运行配置生成逻辑：

- 之前会把 QQ 这种插件渠道错误写进 `openclaw.json.channels`
- 这会导致 `unknown channel id: qqbot` / `unknown channel id: qq`
- 另外，部分 gateway 的 `state_dir/extensions` 没同步到对应 QQ 插件目录

### 4. 旧网关被“自动补回”的原因

根因在运行时配置残留：

- `control_plane/agent_runtime_settings.json` 里之前残留了
  - `active_telegram_instance = "tg-main"`
  - `gw-telegram-tg-main`

程序在归一化运行时配置时，会根据残留的 active 实例/路由/gateway 数据重新补回 gateway。

### 5. 现在两个网关都“拉不起来”的真实原因

最新复查后确认：

- `gw-agent-main` 和 `gw-agent-1` 的 `gateway.cmd` 都存在
- `openclaw gateway start` / `schtasks /Run` 会返回成功
- 计划任务 `LastTaskResult = 0`
- 但两个任务都停留在 `Queued`
- 端口不会因计划任务模式而监听

与此同时，直接前台执行：

```text
openclaw gateway
```

两个 gateway 都能成功监听端口。

结论：

- 坏的不是 OpenClaw 网关本体
- 坏的是 Windows 计划任务模式启动链路
- 桌面端之前依赖的是这条有问题的启动链路

## 已做的代码修复

### 1. 保留真实启动错误

修改 `start_gateway_instance()`：

- 不再吞掉 `ensure_gateway_service_installed()` 失败
- 即使 `gateway start` 返回成功，只要输出里出现 `Gateway service missing` / `schtasks` / `系统找不到指定的文件` 等特征，也按失败回传

### 2. 修正 QQ 插件型渠道的网关配置生成逻辑

修改网关 state_dir 生成流程：

- 插件型渠道不再写入 `openclaw.json.channels`
- `openclaw.json` 只保留 OpenClaw 内置渠道
- 插件渠道改为通过 `channels.json + plugins.entries + extensions` 生效

### 3. 同步 gateway 所需插件目录

新增按需同步逻辑：

- 只把当前 gateway 实际需要的插件目录同步进其 `state_dir/extensions`
- 避免把无关插件一股脑复制进去造成额外警告

### 4. 阻断旧实例自动复活旧 gateway

在运行时配置归一化逻辑中增加校验：

- 只有“真实存在且 enabled 的 Telegram/渠道实例”才允许生成或补回 gateway
- 无效的 `active_telegram_instance` 会被清空
- 无效的 `active_channel_instances` 会被剔除
- 旧 gateway 如果绑定实例已失效，不再保留其旧 `gateway_id`

### 5. 计划任务假成功时自动兜底为直接进程启动

新增兜底逻辑：

- 如果 `gateway start` 返回成功，但几秒内端口没有监听
- 自动改为直接执行 `state_dir/gateway.cmd`
- 从而绕过 Windows 任务计划程序 `Queued` 但不真正拉起监听的问题

## 已做的本机运行态清理
